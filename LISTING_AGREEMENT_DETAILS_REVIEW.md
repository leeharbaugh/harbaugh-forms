## Executive Summary

`listing_agreement_details` is a 152-column table whose **live product role has shrunk to a thin, legacy-URL agreement record**. Its 129 ACTIVE mapped catalog fields (46 on TXR-1101, 83 on TXR-1102) are, in practice, **manual Fill Form fields already** — no current or historical packet field instance has ever been sourced from this table (`source = 'packet'` count: **0**, active and historical).

Three independent breaks make the source family inert:

1. **No packet linkage.** Every current listing packet (5, 13, 19 ACTIVE; 11, 15 DELETED) has `representation_agreement_id = null`. The resolver only loads listing details through the packet → agreement join, so `context.listingAgreementDetails` is always null. The legacy agreement-anchored packet path still exists in the wizard but is labeled "Advanced (legacy)" and has never been used for a listing packet.
2. **Resolver allowlist gaps.** Of 111 ACTIVE fields with a non-null `source_path`, only **31** use paths present in the resolver allowlist (`listing-agreement-field-resolution.ts`). All **80** TXR-1102 lease-expansion paths are missing from the allowlist and could never resolve even if a packet were linked.
3. **Null paths.** **18** ACTIVE fields (17 TXR-1101, 1 TXR-1102) have `source_type = 'listing_agreement_details'` with a null `source_path` — broken by the `sourceTypeRequiresPath` omission (UI save nulls the path) and the 2026-06-24 uppercase-key field generation.

The table holds exactly **one row** (agreement #2, created 2026-06-09, never updated, linked to zero packets). Its ~110 expansion columns carry schema defaults (`NA`, `0`, `false`/`true`, `BrokerBay`, `Dallas/Tarrant`, `FILE_IMMEDIATELY`, …) that predate scoped defaults — they only fire on agreement-UI INSERT and have never injected a value into any packet. Scoped Personal/Organization `field_defaults` plus Fill Form already provide the real behavior for every one of these fields.

**Recommendation:** convert **123** of the 129 mapped fields to `manual_only` (S2), leave **6** fields for Lee decisions (S3), and preserve **no** automatic listing-details sources unconditionally (S1 is empty). Keep the table and its single historical row for now (Option A), stop treating it as a fill source, and fold the schema question into a later cleanup wave with `contract_details`.

## Confirmed Product Model

Consistent with `decisions.md` and both prior audits:

1. Global form templates own PDF structure, placement, and source configuration.
2. Personal and Organization scoped defaults initialize eligible packet field values; they are **not** source mappings.
3. Packets → Fill Form owns form-instance values and manual overrides; Fill Form writes **only** `field_instances`.
4. Ordinary packet open uses `ensure_missing` (insert-only); persisted instances are immutable snapshots.
5. Automatic sources are legitimate only when an independently maintained business object owns the upstream value.
6. A valid PDF placement with `manual_only` and a null source path is a correct, intended state.

**Rule applied here:** if the only user-facing place a value is edited is Fill Form, its details-table source is not an independent automatic business source.
